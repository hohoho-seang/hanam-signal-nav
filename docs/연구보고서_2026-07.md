## Ⅰ. 서론

### 1. 연구 배경

한국의 보행자 신호등은 잔여시간 표시기가 없는 곳이 많고, 있어도 적색 신호의 대기
시간은 표시되지 않는 경우가 대부분이다. 보행자는 신호가 언제 바뀔지 알 수 없어
무리하게 뛰어 건너거나, 반대로 불필요하게 오래 기다리게 된다. 특히 노약자에게는
"지금 출발하면 다 건너기 전에 신호가 바뀌는가"가 안전과 직결된 문제다.

2025년(2학년 마루 아카데미 활동)에 본 연구팀은 하남시 신장동·덕풍동 일대 보행자
신호등 59개의 녹색/적색 신호 길이를 스톱워치로 실측하여, 지도 위에서 신호 상태와
잔여시간을 보여주는 웹앱을 개발했다. 그러나 당시 시스템은 두 가지 근본적 한계를
남겼다. 첫째, 신호의 **주기(녹색+적색 길이)**는 실측했지만 **위상(지금 주기의 어느
지점인지)**을 결정할 기준 시각이 없어, "오전 6시에 모든 신호가 동시에 시작한다"는
검증되지 않은 가정으로 계산했고, 그 결과 실제 신호와 어긋났다. 둘째, 대상이
신장사거리 주변으로 한정되어 실사용 가치가 제한적이었다.

당시 보고서의 제언에는 "현장 관측을 통해 실제 점등 시각을 누적 기록하고 이를
바탕으로 기준 시각을 자동 보정하는 모델", "사용자 참여형(크라우드소싱) 데이터
수집"이 포함되어 있었다. 본 연구(3학년 진로 소모임)는 그 제언을 실제로 구현하고,
대상을 하남시 전역으로 확장하는 것에서 출발했다.

### 2. 연구 목적

1. 실시간 신호개방 API가 제공되지 않는 지자체(하남시)에서, 공공데이터와 사용자
   참여형 보정을 결합해 **실제 신호와 일치하는 보행자 신호 잔여시간 체계**를 구축한다.
2. 신호 대기시간까지 반영한 **보행자 중심 경로 안내(내비게이션)**를 구현한다.
3. 2D 지도의 한계를 보완하는 **3D 씬 및 센서 기반 AR 안내 화면**을 구현한다.
4. 전 과정을 무료 공개 데이터·무료 API만으로, 서버 비용 없이 운영 가능한
   정적 웹앱(PWA)으로 완성해 지속 가능성을 확보한다.

### 3. 연구 문제

- RQ1. 실시간 신호정보가 개방되지 않은 지역에서 보행 신호 잔여시간을 실제와
  일치시키는 것이 가능한가? 가능하다면 어떤 데이터·알고리즘 구조가 필요한가?
- RQ2. 여러 사용자의 부정확한 관측 입력을 통계적으로 결합해 단일 관측보다
  신뢰도 높은 신호 위상 추정치를 만들 수 있는가?
- RQ3. 결제 계정이나 유료 라이선스 없이(학생 신분으로 접근 가능한 자원만으로)
  3D 지도와 AR 내비게이션을 실용적인 수준으로 구현할 수 있는가?

### 4. 연구 범위

- 공간적 범위: 경기도 하남시 전역(보행자 신호등이 설치된 횡단보도 668개소).
  초기 개발·검증은 신장사거리(신장동·덕풍동) 일대에서 수행.
- 내용적 범위: 보행자 신호등 잔여시간 예측, 도보 경로 탐색, 신호 대기 반영
  소요시간 계산, 2D/3D/AR 안내 화면, 사용자 참여형 보정.
- 기술적 범위: 웹 표준 기술(HTML/JS, PWA)과 무료 공개 API만 사용. 네이티브 앱,
  유료 클라우드, 결제 필요 서비스는 제외.
- 기간: 2026년 7월 9일 ~ 7월 15일(개발), 이후 현장 검증 지속 예정.

## Ⅱ. 이론적 배경

### 1. 보행자 교통신호 제어 원리

교차로 신호는 정해진 **주기(cycle)** 안에서 방향별 **현시(phase)**가 순차적으로
반복되는 구조다. 보행자 신호 하나만 놓고 보면 "녹색 g초 → 적색 r초"가 총
주기 T=g+r초로 무한 반복되는 주기 함수로 모델링할 수 있다. 따라서 어떤 시각의
신호 상태는 다음 두 값만 알면 완전히 결정된다.

- **주기 파라미터** (g, r): 신호운영계획에 따라 정해지며 자주 바뀌지 않는다.
- **위상(phase)**: 기준 시각에 주기의 어느 지점을 지나고 있었는가 (0 ≤ φ < T).

이 분해가 본 연구 설계의 핵심이다. 주기는 공공데이터로 확보 가능하지만 위상은
어떤 공개 데이터에도 없다. 그러나 위상은 **단 한 번의 현장 관측**("지금 녹색이고
12초 남았다")으로 확정되며, 신호운영계획이 바뀌기 전까지는 시간 경과에 따라
결정론적으로 전파된다. 즉 "실시간 API 없이도 실시간 일치가 가능한" 이유는 신호가
주기 시스템이기 때문이다.

한편 실제 신호운영은 시간대별 계획(TOD, Time-Of-Day)에 따라 등하교/평시/심야에
다른 주기를 쓸 수 있어, 하나의 (g, r)로 하루 전체를 설명하지 못할 수 있다.
본 연구는 이를 위해 보정 입력에 시간대 태그를 함께 수집한다(Ⅲ-5).

### 2. 지도 API와 위치 기반 서비스

- **Leaflet + OpenStreetMap(OSM)**: 오픈소스 지도 라이브러리와 공개 지도 타일.
  API 키나 도메인 등록이 필요 없어 배포 주소가 바뀌어도 제약이 없다.
- **Tmap Open API(SK)**: 보행자 경로 탐색, 장소(POI) 검색, 역지오코딩을 무료
  제공하며, CORS를 지원해 서버 없이 브라우저에서 직접 호출할 수 있다(본 연구가
  서버리스 구조를 유지할 수 있었던 핵심 조건).
- **Geolocation API / DeviceOrientation API**: 브라우저 표준 위치·방향 센서 접근.
  iOS는 방향 센서 접근에 명시적 사용자 제스처 안에서의 권한 요청을 요구한다.
- 거리 계산은 하버사인(haversine) 공식, 두 지점 간 방위각은 대권 방위각 공식을
  사용했다.

### 3. PWA와 웹앱 기술

PWA(Progressive Web App)는 웹앱을 홈 화면에 설치해 네이티브 앱처럼 쓸 수 있게
하는 기술로, manifest와 Service Worker로 구성된다. Service Worker의 캐시 전략은
신중히 선택해야 한다. 본 연구 초기에는 cache-first 전략을 썼는데, 배포 후에도
재방문자가 캐시된 옛 데이터를 계속 보는 문제가 실제로 발생했다(신호 데이터를
갱신했는데 이전 버전이 표시됨). 이를 자주 바뀌는 자원(신호 데이터, 앱 코드)은
network-first, 거의 안 바뀌는 자원(라이브러리, 아이콘)만 cache-first로 나누는
혼합 전략으로 해결했다(Ⅲ-9).

### 4. 실시간 데이터베이스

Firebase Firestore는 서버 없이 브라우저에서 직접 읽고 쓸 수 있는 클라우드
데이터베이스로, `onSnapshot` 구독을 통해 데이터 변경이 모든 접속자에게 즉시
전파된다(실측 기준 1초 이내). 여러 사용자가 동시에 같은 문서를 갱신할 때의
경합은 `runTransaction`(원자적 읽기-계산-쓰기)으로 처리했다. 보안은 Firestore
보안규칙으로 담당한다: 누구나 읽을 수 있으나, 쓰기는 정해진 스키마(상태·잔여초
필드와 값 범위)를 만족할 때만 허용해 로그인 없는 참여와 데이터 무결성을 양립시켰다.

### 5. 순환통계(circular statistics)

신호 위상은 0~T초에서 감기는(wrap-around) 순환량이므로 산술평균을 쓸 수 없다.
예컨대 T=360초에서 위상 5초와 355초의 산술평균은 180초로 완전히 틀리지만, 실제
두 관측은 모두 "0초 부근"을 가리킨다. 순환평균은 각 위상을 단위원 위의 각도로
변환한 뒤 벡터 합의 방향을 취한다:

φ̄ = atan2( Σ sin(2πφᵢ/T), Σ cos(2πφᵢ/T) ) × T / 2π

또한 서로 다른 시각에 관측된 위상들을 결합하려면 각 관측을 공통 기준 시각으로
**시간 전파**(φ' = (φ + Δt) mod T)한 뒤 평균해야 한다. 이상치는 기존 평균과의
순환거리(짧은 쪽 호의 길이)가 주기의 30%를 넘는 입력을 배제하는 방식으로
걸러냈다. 구현 검증 수치는 Ⅳ-3에 제시한다.

### 6. 3차원 지도와 증강 현실

- **Three.js**: WebGL 기반 3D 렌더링 라이브러리. OSM 건물 외곽선(footprint)을
  압출(extrude)하면 키·모양이 실제 배치를 반영하는 간이 3D 도시 모델을 만들 수 있다.
- **WebXR vs 센서 기반 AR**: WebXR은 SLAM 기반 정밀 공간추적 AR 표준이지만
  iOS Safari가 지원하지 않는다. 대안인 센서 기반 AR는 카메라 영상 위에 나침반
  (방위각)과 GPS(위치)로 계산한 3D 오버레이를 합성하는 방식으로, 정밀도는
  낮지만 iOS·Android 모두에서 동작한다. 본 연구는 후자를 채택했다(Ⅲ-8).
- 센서 원신호(나침반·GPS)는 노이즈가 커서 그대로 화면에 반영하면 떨림·튐이
  발생한다. 목표값과 표시값을 분리하고 매 프레임 일정 비율씩 수렴시키는
  지수 평활(exponential smoothing)로 해결했으며, 방위각은 순환량이므로 최단
  방향 보간(angle lerp)을 사용했다.

### 7. 선행연구 고찰

- **작년 연구(2025, 마루 아카데미)**: 신장동·덕풍동 59개 신호 실측, 네이버지도
  기반 표시 앱. 성과는 실측 주기 데이터 자체와 PWA 구현 경험, 한계는 위상
  미동기화 오차와 좁은 대상 범위. 본 연구의 직접적 토대다.
- **경찰청·도로교통공단 실시간 신호정보 개방 사업**: 차량 내비게이션(카카오내비,
  T맵, 현대차그룹 내비)에 교차로 신호 잔여시간을 제공하는 국가 사업. 2026년 현재
  서울, 강릉(전 교차로), 천안(69개), 의왕, 대전, 고양 등으로 확대 중이나
  **하남시는 미포함**이며, 데이터 연계도 내비게이션 사업자 대상(B2B)이라 일반
  개발자가 쓸 수 있는 공개 API가 아니다. 도시교통정보센터(UTIC)의 신호개방
  데이터도 인천·대구 2개 지역의 운영계획(TOD)을 하루 1회 배포하는 수준으로
  실시간 잔여시간과는 거리가 있다. 이 조사가 "공공 실시간 API 대기"가 아니라
  "자체 체계 구축"으로 방향을 정한 근거다.
- **전국횡단보도표준데이터(공공데이터포털)**: 지자체가 등록하는 횡단보도 표준
  데이터로 위도·경도, 보행자신호등유무와 함께 **녹색신호시간·적색신호시간**
  필드를 포함한다. 본 연구가 주기 데이터의 공식 출처로 활용했다(Ⅲ-2).

## Ⅲ. 연구 과정 및 시스템 개발

### 1. 기존 연구 분석 및 문제 발견

작년 시스템의 코드를 분석해 다음 문제를 확인했다.

1. **위상 가정 오류**: 잔여시간을 "오전 6시 기준 경과시간 mod 주기"로 계산 —
   모든 신호가 매일 오전 6시에 동시에 초기화된다는 가정인데 근거가 없다.
2. **경로 안내 미완성**: 경로 탐색 코드가 좌표 하드코딩, 차량용 API 호출,
   응답 필드명 오타(trapopimal)로 실제로 동작하지 않는 상태였다.
3. **데이터 중복**: 59개 항목 중 동일 지점 중복 입력이 다수였고, 같은 좌표에
   서로 크게 다른 주기가 기록된 지점이 4곳 있었다(측정 오차인지 시간대별
   운영 차이인지 판단 불가).
4. **배포 인프라 소멸**: 기존 호스팅(dothome) 계정이 만료되어 접근 불가.

### 2. 공공데이터 분석 및 데이터 전처리

**실시간 API 재조사(7/14~15)**: 서론의 결론(하남 미제공)을 최신 기사·공식 자료로

### 3. 시스템 구조 재설계

초기에는 작년과 같은 PHP 호스팅 재구축을 시도했으나 신규 계정의 FTP 업로드가
반복 실패해, **GitHub Pages 정적 호스팅**으로 전면 전환했다. 이 결정이 구조를
크게 단순화했다:

- 서버 코드(PHP) 전체 폐기 → 유지비 0원, 배포는 git push 한 번.
- 경로 API 프록시 불필요 확인(Tmap이 CORS 지원 → 브라우저 직접 호출).
- 보정 데이터 저장은 Firebase Firestore(무료 티어)로 대체.
- 지도는 도메인 등록이 필요한 네이버 지도에서 Leaflet+OSM으로 교체 —
  이후 배포 주소가 바뀌어도 재등록 절차가 없다.

최종 구조: 정적 파일(HTML/JS/JSON) + 외부 무료 API 4종(Tmap, Firestore,
OSM 타일, OSM Overpass). 시스템 구조도는 부록 F.

### 4. 보행자 내비게이션 구현

- **경로 탐색**: Tmap 보행자 경로 API. 출발·도착 좌표로 도보 경로(GeoJSON)를
  받아 지도에 표시한다.
- **경로-신호 매칭**: 경로 좌표열과 신호 좌표의 최근접 거리가 15m 이내인 신호를
  경로상 신호로 판정하고, 같은 교차로의 중복(15m 이내 상호 근접)은 하나로
  묶는다. 매칭 반경은 원래 40m였으나 하남 전체 데이터(696개)로 밀도가 높아진 뒤
  실제로 건너지 않는 옆 횡단보도까지 잡히는 과대 매칭이 확인되어 15m로 조정했다
  (같은 495m 시험 경로에서 매칭 12→8개).
- **누적 지연(cascading delay) 소요시간 계산**: 보행 속도(기본 1.2m/s, 조절
  가능)로 각 신호 도착 예상 시각을 구하고, 그 시각의 신호 상태를 예측해 적색이면
  잔여 대기시간을 가산한다. 중요한 것은 이 대기가 **다음 신호들의 도착 시각을
  뒤로 미룬다**는 점이다 — 앞 신호에서 40초 기다리면 뒤 신호에는 40초 늦게
  도착하므로 예측 상태가 달라진다. 이를 순차 전파해 "신호 대기까지 반영한 총
  소요시간"을 산출한다.
- **실시간 추적**: 내비게이션 시작 시 GPS(watchPosition)로 위치를 따라가며,
  2D 지도에는 진행 방향 화살표 마커가 이동하고 다음 신호 접근 시(60m 이내)
  잔여시간 HUD가 자동 표시된다.

### 5. 실시간 보정 시스템 개발

작년 제언(크라우드소싱 보정)의 실제 구현이다.

- **1탭 위상 보정**: 사용자가 현장에서 "지금 이 신호는 녹색이고 12초 남음"을
  입력하면, 그 관측이 위상 앵커가 되어 이후 잔여시간이 실제와 일치하게 된다.
  주기는 이미 공식 데이터로 알고 있으므로 관측 1회로 충분하다.
- **실시간 공유**: 보정은 Firestore에 저장되어 모든 사용자에게 1초 이내에
  반영된다(한 명의 보정이 전체의 정확도를 올리는 구조).
- **순환통계 정제**: 여러 사용자의 입력이 쌓이면 각 입력을 위상으로 환산해
  시간 전파 후 순환평균으로 결합하고, 기존 평균과 주기의 30% 이상 벗어나는
  입력은 이상치로 배제한다(잘못된 입력 1건이 전체를 오염시키지 않음). 동시
  제출 경합은 Firestore 트랜잭션으로 원자 처리한다.
- **참여 유도 UX**: 앱 접속 시 GPS 반경 100m 내에 미보정 신호가 있으면
  "지금 이 신호가 보이시나요?" 배너를 자동 표시하고, 큰 녹/적 버튼과 슬라이더로
  현장에서 몇 초 만에 입력을 마칠 수 있게 했다(작년 연구에서 참여율이 사실상
  0이었던 원인을 '입력 진입장벽'으로 진단한 결과).
- **시간대 태그**: 모든 보정 입력에 등하교/낮/저녁/심야 태그를 함께 저장한다.
  당장은 수집만 하고, 데이터가 축적되면 시간대별(TOD) 주기 전환 대응에 쓴다.

### 6. 검색 기능 및 사용자 인터페이스 개선

- **장소 검색 자동완성**: 출발지·도착지 입력창에 2글자 이상 입력하면(300ms
  디바운스) Tmap POI 검색으로 실존 장소를 추천하고, 선택 시 좌표가 확정된다.
  임의 텍스트 입력의 "실존하지 않는 주소" 문제를 원천 차단한다.
- **역지오코딩**: 지도 클릭이나 "내 위치" 버튼으로 지정한 지점도 주소로 변환해
  입력창에 자동 표시한다. 세 가지 입력 경로(검색/지도클릭/GPS)가 동일한 내부
  처리를 공유한다.
- **마커 클러스터링**: 신호 696개를 개별 표시하면 지도가 마커로 뒤덮이므로
  축소 상태에서는 클러스터로 묶고 확대(줌 17+)하면 개별 표시한다. 매초 아이콘
  갱신은 상태가 실제로 바뀐 마커만 수행하도록 최적화했다(696개 전체 갱신 루프
  실측 3.9ms).

### 7. Three.js 기반 3D 내비게이션 구현

처음에는 실사진 기반(카카오 로드뷰)을 시도했으나, 로드뷰는 이산적인 촬영
지점의 파노라마라 연속적인 경로 추적에 부적합했다(파노라마 간 점프 발생).
이를 폐기하고 Three.js로 직접 3D 씬을 구축했다.

- **건물**: OSM Overpass API에서 경로 주변 건물 외곽선을 받아 층수·높이 태그
  기반으로 압출. 용도별 7색(아파트/학교/상가/병원 등)으로 구분하고, 이름이
  등록된 건물은 지붕 위에 이름표를 표시해 "어느 건물인지" 식별 가능하게 했다.
- **도로**: 도로 중심선을 도로 등급별 표준 폭의 리본(ribbon) 지오메트리로
  렌더링(차도/보행로 색 구분).
- **경로**: 굵은 튜브로 그리면 사용자가 "경로를 뚫고 지나가는" 부자연스러움이
  있어, 바닥에 붙는 얇은 리본(페인트 표시 느낌)으로 표현했다. 또한 경로와
  건물의 데이터 출처가 달라(Tmap vs OSM) 정합이 완벽하지 않으므로, 경로가
  건물에 가려 보이지 않는 것을 막기 위해 깊이 검사를 끄고 항상 위에 그린다
  (상용 지도 AR과 같은 방식).
- **카메라**: 경로 계산 직후에는 출발점에 정지 상태로 두고, 실제 이동은 GPS
  추적(내비게이션 시작) 또는 명시적 미리보기 버튼으로만 움직인다 — 초기
  구현에서 자동 재생을 넣었다가 "실제 움직임과 무관하게 화면이 움직인다"는
  문제로 제거했다.

### 8. 센서 기반 AR 구현

3D 씬만으로는 실제 주변 환경과의 대응이 어렵다는 판단에 따라, 카메라 영상 위에
3D 씬을 겹치는 AR 모드를 추가했다.

- **구성**: 후면 카메라 영상을 배경으로 깔고, Three.js 렌더러를 투명 배경으로
  전환해 경로 리본·신호 기둥·건물을 그 위에 합성한다. 위치는 GPS, 시선 방향은
  나침반 센서(iOS webkitCompassHeading / Android deviceorientationabsolute)가 담당한다.
- **방향 처리의 시행착오**: ①나침반+기울임(beta/gamma) 3축 반영 → 실기기에서
  상하 미동작 ②기울임 제거+고정 하향각 → 실기기에서 좌우 반전·화면 불안정
  ③**나침반 1축(좌우)만 반영하는 최초 단순 구현으로 회귀** — 실기기(iPhone)에서
  가장 안정적임을 확인하고 확정했다. "센서 자유도를 높이는 것이 이론상 정확해
  보여도 실사용 조작감을 해칠 수 있다"는 것이 핵심 교훈이다(부록 C).
- **센서 스무딩**: 나침반 노이즈와 GPS 점프가 화면 떨림·튐으로 직결되는 문제를,
  센서 콜백은 목표값만 갱신하고 실제 카메라는 매 프레임 일정 비율(방위 15%,
  위치 8%)씩 수렴시키는 구조로 해결했다. 방위각은 최단 방향 보간(350°→10°는
  20°만 회전)을 적용했다. 실기기에서 "부드럽게 동작함"을 확인했다.

### 9. 시스템 통합 및 최종 검증

- **검증 방법론**: 지도 SDK의 도메인 제한 등으로 로컬 환경에서 화면 전체를
  띄우기 어려운 경우가 많아, 브라우저 콘솔에서 **내부 함수를 직접 호출해 반환값을
  수기 계산과 대조**하는 방식을 표준 절차로 삼았다. 예: 순환평균 함수에 합성
  샘플 투입, 경로-신호 매칭 함수에 실데이터 투입, 누적 지연 계산의 단계별 값
  검증. 배포 확인은 브라우저 캐시를 우회해 원본 서버 응답을 직접 조회했다.
- **교차검증(중요 결과)**: 작년 스톱워치 실측과 하남시 공식 데이터가 겹치는
  15개 지점을 비교한 결과 다수 지점에서 **초 단위까지 정확히 일치**했다
  (Ⅳ-3 표 참조). 이는 작년 실측 방법론의 유효성을 공식 데이터로 입증하는
  동시에, 공식 데이터를 신뢰할 근거가 된다.
- **성능 검증**: 696개 신호 로드 상태에서 마커 갱신 루프 3.9ms, 경로 탐색+매칭
  +ETA 계산 약 0.2초(API 왕복 포함) — 실사용에 지장 없는 수준.

## Ⅴ. 결론 및 제언

### 1. 결론

본 연구는 실시간 신호정보가 개방되지 않은 하남시에서, "신호는 주기 시스템이므로
**주기(공공데이터) × 위상(1회 현장 관측)**으로 분해하면 실시간 API 없이도 실제와
일치하는 잔여시간을 계산할 수 있다"는 가설을 시스템으로 구현했다(RQ1). 여러
사용자의 관측을 순환평균과 이상치 배제로 결합하는 정제 알고리즘을 구현·검증했고
(RQ2), 무료 자원만으로 3D·AR 안내를 포함한 전체 시스템을 완성했다(RQ3).
특히 작년 스톱워치 실측과 올해 공식 데이터의 초 단위 교차 일치는 2년에 걸친
연구의 방법론적 연속성을 보여주는 결과다.

### 2. 한계

1. **위상 보정의 현장 실증 미완**: 체계는 완성됐으나 "보정 후 수 시간~수일이
   지나도 실제 신호와 일치하는가"의 현장 검증이 남아 있다(신호제어기 시계
   드리프트, 감응식 신호 등 변수 존재 가능).
2. **TOD(시간대별 운영) 미대응**: 하나의 주기로 하루 전체를 가정한다. 등하교
   시간대에 다른 주기를 쓰는 교차로는 해당 시간대에 오차가 생길 수 있다
   (시간대 태그 수집으로 대응 준비 중).
3. **경로-신호 매칭의 한계**: 15m 반경 매칭은 실제로 건너는 신호와 옆을
   지나가기만 하는 신호를 완전히 구분하지 못한다.
4. **3D 정밀도**: OSM 건물 데이터의 한국 커버리지 한계로 실제 건물과 형태가
   다를 수 있다.
5. **AR 정밀도**: SLAM이 아닌 나침반+GPS 기반이라 오버레이가 수 미터 어긋날
   수 있으며, 보행 중 동작 검증이 남아 있다.

### 3. 제언

1. **지자체·경찰청에 대한 제언**: 실시간 신호개방 사업의 하남시 확대, 그리고
   내비게이션 사업자 외 일반 개발자·연구자용 공개 API 제공을 제안한다. 본
   연구처럼 공공데이터(주기)만으로도 상당한 서비스가 가능함은 확인됐으나,
   위상 정보가 개방되면 크라우드소싱 없이 즉시 전 지점 실시간 일치가 가능하다.
2. **후속 연구**: ①현장 실증(보정 후 지속 일치 확인, 시간대별 오차 측정)
   ②주기 미상 지점을 위한 "3탭 주기 실측 모드"(녹색 시작→적색 시작→다음 녹색
   시작 탭으로 주기와 위상을 동시 확보) ③경로가 횡단보도를 실제로 가로지르는지
   기하 판정 ④신호 대기 최소 경로 추천(경로 후보별 총 소요시간 비교).

### 부록 B. 연구 노트 (주요 의사결정 기록)

1. **"실시간"의 재정의**: 실시간 API가 없다 = 실시간 불가능이 아니라, 신호가
   결정론적 주기 시스템이므로 "주기 × 위상"만 확보하면 로컬 계산으로 실시간
   일치가 가능하다는 관점 전환이 이번 연구의 출발점.
2. **크라우드소싱 입력의 단위 선택**: "지금 상태+잔여초"(1탭)를 입력 단위로
   정한 이유 — 주기를 이미 알기 때문에 위상 확정에 필요한 최소 정보이며,
   현장에서 신호등 잔여 표시를 보고 즉시 입력 가능하다.
3. **이상치 임계 30%**: 주기의 30%(예: T=140이면 42초) 이상 벗어난 입력을
   배제 — 시계 오차·입력 실수는 걸러내되, 정상 관측의 자연 오차(수 초)는
   통과시키는 수준으로 설정.
4. **기존 실측 44개의 id 보존 원칙**: 데이터 정리·확장 시 식별자를 재배정하면

```javascript
function circularMeanPhase(phases, totalCycle) {
  let sumSin = 0, sumCos = 0;
  phases.forEach(p => {
    const angle = (p / totalCycle) * 2 * Math.PI;
    sumSin += Math.sin(angle);
    sumCos += Math.cos(angle);
  });
  const meanAngle = Math.atan2(sumSin, sumCos);
  return wrapPhase((meanAngle / (2 * Math.PI)) * totalCycle, totalCycle);
}
```

  const d = ((b - a + 540) % 360) - 180; // -180~+180로 정규화된 최단 차이
  return (a + d * t + 360) % 360;
