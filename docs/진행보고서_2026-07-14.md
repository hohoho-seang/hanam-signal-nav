## 1. 지금까지 한 내용

### 1.1 배포 인프라
- 처음엔 dothome(PHP) 호스팅으로 계획했으나 FTP 업로드가 계속 실패해 **GitHub Pages 정적 사이트**로 전환
- 이 결정 덕분에 서버 로직(PHP)을 전부 걷어내고, 모든 API를 브라우저에서 직접 호출하는 구조로 단순화됨
- PWA(홈 화면에 앱처럼 추가 가능) 구성, Service Worker 캐시 무효화 버그를 발견해 수정(배포해도 재방문자가 옛날 데이터를 계속 보던 문제)

### 1.2 지도 · 경로
- 지도: Naver Maps → **Leaflet + OpenStreetMap**으로 교체 (키/도메인 등록 절차 없이 어디서나 바로 동작)
- 경로 탐색: **Tmap 보행자 경로 API**를 브라우저에서 직접 호출
- 경로상의 신호등을 자동으로 매칭해서, 빨간불 예측 시 대기시간을 뒤따르는 신호에도 누적 반영(신호 대기까지 포함한 총 소요시간 계산)

### 1.3 신호 잔여시간 정확도
- **문제**: 작년 실측값은 정확하지만 "언제부터 신호가 시작됐는지" 기준 시각이 없어서, 시간이 지날수록 실제 신호와 어긋남
- **해결**: 사용자가 현장에서 "지금 신호 상태"를 입력하면(1탭) 그 시점을 기준으로 정확히 역산하는 **실측 보정 시스템** 구축
- 이 보정값을 **Firebase(Firestore)로 실시간 공유** — 한 사람이 입력하면 다른 모든 사용자에게 반영됨
- 여러 사람의 입력이 쌓이면 **순환통계(circular mean)**로 이상치를 걸러내며 점점 더 정확해지는 구조
- 지도 접속 시 GPS 반경 100m 내 미보정 신호가 있으면 자동으로 "지금 신호 알려주세요" 배너를 띄워 참여 장벽을 낮춤
- `signals.json` 데이터 정리: 59개 → 44개(중복 병합), 애매하게 값이 다른 4곳(신장동 422-10, 621, 695, 571)은 삭제하지 않고 두 후보 값을 모두 보존해둠(향후 실측으로 확정 예정)

### 1.4 3D 내비게이션
- Google Street View(결제계정 필요)와 Kakao 로드뷰(사진이 이산적이라 경로 추적에 부적합) 모두 검토 후 폐기
- 최종적으로 **Three.js로 직접 3D 씬 구성**: 실제 건물(OpenStreetMap 데이터), 도로, 경로선을 전부 좌표 기반으로 렌더링
- GPS로 실시간 위치를 추적하며 카메라가 경로를 따라가는 내비게이션 구현
- 3D/2D 화면을 서로 바꿔볼 수 있는 PiP(화면 속 화면) UI

### 1.5 AR 모드
- WebXR(정밀 공간추적 AR)은 iOS Safari가 지원하지 않아서 배제
- 대신 **카메라 화면 + 나침반 + GPS**를 조합한 센서 기반 AR 구현 — 실제 카메라 화면 위에 경로와 신호 정보가 겹쳐 보임
- 실기기 테스트를 여러 차례 거쳐 **나침반 좌우 회전만 반영하는 가장 단순한 방식이 가장 안정적**이라는 결론에 도달(위아래 기울임 반영은 시도했으나 오히려 조작감이 나빠져서 최종적으로 제외)

### 1.6 검색 기반 출발지/도착지 설정
- 기존엔 지도를 직접 클릭해야만 출발/도착지를 정할 수 있었음
- **Tmap 장소 검색 API**로 실제 존재하는 장소만 자동완성 추천되도록 개선(진짜 내비게이션 앱처럼 동작)
- 지도 클릭이나 "내 위치" 버튼으로 찍은 지점도 자동으로 주소가 입력창에 채워짐(역지오코딩)

## 3. 직접 하셔야 할 것 — 현장 실측 사진

정확도를 높이려면 실제 현장 데이터가 필요합니다. 간단히 정리하면:

1. **애매한 4개 지점 우선** (신장동 422-10 · 621 · 695 · 571) — 각각 다른 시간대(등교/평시/하교/저녁)에 방문해서 "지금 신호 상태 + 남은 초"를 앱에 입력하거나 사진/영상으로 기록
2. **나머지 신호들도 틈틈이** — 지나가는 길에 앱의 "지금 신호 알려주기" 배너나 마커 클릭으로 1탭 입력
3. **발표/보고서용 사진** — 실제 신호등, 교차로 전경, AR 화면 스크린샷 몇 장 있으면 결과물 설명할 때 좋음

촬영/기록 시 **날짜·시각**만 같이 남겨두시면 나중에 데이터 정리할 때 큰 도움이 됩니다.
